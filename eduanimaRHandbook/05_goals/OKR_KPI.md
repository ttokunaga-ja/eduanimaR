# OKR / KPI

本ドキュメントは、01章で定義したミッション／原則（学習支援・データ最小化・厳格なアクセス制御・透明性）を、測定可能な目標に落とし込む。

スコープ:
- 本ドキュメントで定義する: 目標、測定方法、到達基準、レビューサイクル。
- 本ドキュメントで定義しない: 実装（DB/クラウド/フレームワーク/アーキテクチャ）、具体UI、個別のプロジェクト管理（チケット、スプリント）。

前提（変えない要求）:
- 提供形態: Chrome拡張機能 + Webアプリ。
- 認証: SSO（OAuth）を用いる。
- 認可: ユーザーごとのアクセス制限を厳格に実施し、導線（拡張/WEB）に依存しない。
- 共有: 将来「科目内（授業内）グループ共有」を想定するが、共有対象は原則「科目の資料セット」に限定し、質問履歴や個人の学習ログ等は共有しない。

## North Star
学習者が「探す」より「理解する」時間を増やせていること。

North Star Metric（一次指標）:
- 主要タスク完了時間（資料から根拠箇所に到達するまでの時間）の短縮

補助指標（北極星の因果を支える）:
- 根拠提示率（回答/提案が“参照箇所”を伴う割合）
- 目標行動明確化率（次に何をすべきかが明確になった、と自己申告される割合）

## 計測の原則（KPI設計ガードレール）
- 個人情報最小化: 指標のために不要なデータを増やさない。
- 共有境界の尊重: 個人データ（質問履歴・学習ログ等）を“共有前提の指標”にしない。
- 監査可能性: 認証・認可・アクセス制御に関わる指標は、後から検証できる。

## OKR（2026 H1 / 初期リリース〜安定運用フェーズ）

### O1. Trust & Security（信頼と安全）を最優先で成立させる
Key Results:
- KR1: 重大インシデント（他ユーザーのデータが閲覧できる、権限逸脱、認証不備）= 0
- KR2: 認可の“意図しない許可”を検知できる監査観点が定義され、定期レビューが実施されている（最低月1回）
- KR3: アカウント関連の問い合わせ（ログインできない/権限が違う等）が、週次で追跡され、改善サイクルに入っている

主要KPI（例）:
- 認可拒否イベント率（異常増加がないこと）
- 権限境界テストのカバレッジ（観点数ベースで管理）
- セキュリティ・プライバシー問い合わせ件数 / 週

### O2. 学習支援として“使える”体験を提供する（自動正解提示を目的化しない）
Key Results:
- KR1: 主要ユースケース（資料探索、要点抽出、学習ロードマップ作成）で、ユーザーが「役に立った」と評価する割合が基準を超える（初期は定性→定量化）
- KR2: 生成物が「参照箇所（根拠）」を伴う割合が一定水準を超える
- KR3: 誤情報/根拠不明の指摘が、件数として追跡され、改善サイクルが回っている

主要KPI（例）:
- タスク成功率（目的タスクを完了できた割合）
- 参照箇所提示率
- “役に立った”評価（5段階/二択など、運用可能な形式で）
- 誤情報報告率（セッションあたり）

### O3. Chrome拡張 + Webアプリで一貫した利用を成立させる
Key Results:
- KR1: 拡張機能とWebアプリで、同一アカウント・同一権限で利用できる（導線による権限差分がない）
- KR2: SSO（OAuth）ログインの成功率が十分高く、初回導入の離脱が低い

主要KPI（例）:
- ログイン成功率 / 失敗理由の内訳
- 初回アクティベーション率（インストール/初回ログイン/初回利用の到達率）
- 拡張→Web（またはその逆）の遷移成功率（必要な場合）

### O4. 資料の取り込み〜検索補助が、学習支援として機能する
意図:
- 事前OCR（Text化）は、エージェントによる資料探索・検索補助のため。

Key Results:
- KR1: 取り込み対象資料のうち、検索に利用可能な“テキスト化済み割合”が基準を超える
- KR2: 検索ヒットの再現率/適合率が、運用上の最低ラインを満たす（初期はスポット検証→定点観測）

主要KPI（例）:
- OCR成功率（失敗理由の分類込み）
- テキスト化までのリードタイム（アップロード→検索可能）
- 検索0件率（クエリあたり）

### O5. 運用可能性（Reliability & Cost）を確立する
Key Results:
- KR1: 主要機能の可用性が一定水準を満たす（定義と計測方法を確立する）
- KR2: 月次のコストが追跡可能で、コスト主要因が説明できる
- KR3: 障害時の検知→切り分け→復旧の手順が整備され、訓練（机上/簡易）を実施できる

主要KPI（例）:
- 可用性（SLO/エラーバジェットの考え方を採用するかは別途）
- p95 レイテンシ（主要API/主要タスク）
- 月次コスト（分類: 認証、推論、ストレージ、DB、ネットワーク等）

## 共有機能（将来）に関する目標の置き方
“いつかやる”を定義するが、今すぐの実装を約束しない。

将来OKR候補（時期未定）:
- 科目内グループ共有を提供しても、個人データ（質問履歴・学習ログ等）が共有されないことがユーザーに理解でき、監査可能である

将来KPI候補:
- 共有設定の誤解率（「共有したつもり/してないつもり」問い合わせの割合）
- 共有範囲の変更履歴が追跡できる割合

## レビュー運用
- 月次: KPIレビュー（異常検知と改善アクションの決定）
- 四半期: OKRレビュー（継続/変更/廃止、KPI定義の見直し）
